# Evaluation Recycling (Amortized Signal Extraction)

## The Problem

Every iteration, GEPA evaluates a candidate on a minibatch (3 examples) with full traces. The traces — the rich diagnostic information, the ASI (Actionable Side Information) — are used for one reflection and then discarded. The scores are cached in `EvaluationCache`, but the traces (error messages, expected vs actual, execution logs) are thrown away.

Over 50 iterations, GEPA generates ~150 trace evaluations. Each contains detailed information about how a specific candidate behaved on a specific example. This is an enormous amount of diagnostic signal that's used exactly once and forgotten.

The reflection LLM sees 3 traces per iteration. It makes a diagnosis from 3 data points. Meanwhile, there are potentially 10-20 other traces *of the same candidate on different examples* sitting in recent history, never shown to the reflection LLM.

## The Fix: Store Traces, Retrieve at Reflection Time

Store all evaluation traces in a structured trace database. When reflecting, retrieve traces not just from the current minibatch but from recent iterations — specifically, traces of the *current candidate* (or its recent ancestors) on *different* examples that scored below threshold.

```
Trace DB schema:
  (candidate_hash, example_id, score, trace_text, iteration)

At reflection time:
  1. Get 3 traces from current minibatch — standard GEPA behavior
  2. Query trace DB: recent traces of this candidate's lineage on OTHER examples
     that scored below threshold (i.e., other known failures)
  3. Include top 3-5 retrieved failure traces in the reflective dataset

The reflection LLM now sees: 3 fresh traces + 3-5 historical traces = 6-8 traces total
```

## Why This Is Better Than Just Increasing Minibatch Size

Increasing the minibatch from 3 to 8 costs 5 extra evaluation calls per iteration — expensive. Recycling old traces costs zero evaluation calls — the traces are already computed and sitting in memory.

You get the diagnostic breadth of an 8-example minibatch at the evaluation cost of a 3-example minibatch.

The trade-off: recycled traces may be slightly stale. But "slightly stale" is much better than "nonexistent."

## The Staleness Concern and Why It's Manageable

Traces from 5 iterations ago were generated by an ancestor candidate, not the current one. If the candidate has changed significantly, old traces may describe failures that no longer exist (or miss new failures that have appeared).

However:
- With ~30% acceptance rate, the candidate changes every ~3 iterations on average. Traces from 1-2 iterations back are almost always from the same candidate.
- Even for traces from an older ancestor: if the ancestor failed on example X, and the descendant hasn't been explicitly fixed for X, the descendant probably still fails on X. Text mutations are typically small — one or two sentences change per accepted mutation.
- Recycled traces from failures are more robust to staleness than traces from successes. A candidate that failed on X probably still fails on X unless specifically fixed. But a candidate that succeeded on X might regress after mutation.

**Policy**: Only recycle failure traces (score < threshold). Don't recycle success traces (they're more fragile to staleness).

## Where It Intervenes

### New: TraceDatabase on GEPAState

```python
@dataclass
class StoredTrace:
    candidate_hash: str
    candidate_idx: int
    example_id: DataId
    score: float
    trace_text: str  # the formatted side_info / ASI
    iteration: int

class TraceDatabase:
    def __init__(self):
        self.traces: list[StoredTrace] = []

    def store(self, candidate_hash: str, candidate_idx: int,
              example_id: DataId, score: float, trace_text: str,
              iteration: int):
        self.traces.append(StoredTrace(
            candidate_hash, candidate_idx, example_id,
            score, trace_text, iteration
        ))

    def retrieve_failures(
        self,
        candidate_idx: int,
        exclude_ids: set[DataId],  # current minibatch — don't duplicate
        state: GEPAState,
        max_lineage_depth: int = 3,
        score_threshold: float = 0.5,
        k: int = 5,
    ) -> list[StoredTrace]:
        """Retrieve recent failure traces from this candidate's lineage."""
        # Walk lineage: current candidate, parent, grandparent...
        lineage = set()
        idx = candidate_idx
        for _ in range(max_lineage_depth):
            lineage.add(idx)
            parent = state.parent_program_for_candidate.get(idx)
            if parent is None:
                break
            idx = parent

        # Filter: in lineage, below threshold, not in current minibatch
        candidates = [
            t for t in self.traces
            if t.candidate_idx in lineage
            and t.score < score_threshold
            and t.example_id not in exclude_ids
        ]

        # Sort by recency, take top k
        candidates.sort(key=lambda t: t.iteration, reverse=True)
        return candidates[:k]
```

## Interaction with Existing Systems

### Evaluation Cache (`EvaluationCache`)
The evaluation cache stores `(candidate_hash, example_id) → (output, score, objective_scores)`. It avoids redundant evaluation. The trace database stores the *traces* (diagnostic text) — information the cache doesn't keep. They're complementary:
- Cache answers: "What score did this candidate get on this example?"
- Trace DB answers: "What went wrong when this candidate tried this example?"

### Reflective Dataset Format
The reflective dataset is built by `adapter.make_reflective_dataset()`. Historical traces need to be formatted in the same structure so the reflection LLM can process them uniformly. Mark historical traces with a note: "[From evaluation 3 iterations ago]" so the LLM knows they're not fresh.

### Prompt Length
Adding 4-5 recycled traces increases the reflection prompt by ~500-1000 tokens. This is well within typical context limits (4K-8K tokens for the reflection call). If context is tight, reduce K or truncate trace text.

## Design Decisions

**Max lineage depth.** How far back in the parent chain to look for traces. Depth 1 = only current candidate. Depth 3 = current + parent + grandparent. With ~30% acceptance rate and mutations changing 1-2 sentences, depth 3 covers the last ~9 iterations and the candidate text is very similar throughout.

**Score threshold for "failure."** Which traces count as failures worth recycling. Use an adaptive threshold: the running median score across recent evaluations. This adapts as the candidate improves — early in optimization, most things are failures; late, only the truly hard examples.

**Deduplication.** Don't show two traces of the same example from different ancestors. Keep only the most recent trace per example_id.

**Memory management.** Over a long run, the trace DB grows. Prune traces older than a threshold (e.g., 30 iterations) or from candidates no longer on the Pareto front.

## What Can Go Wrong

**Stale traces mislead the reflection LLM.** A trace from 5 iterations ago shows the candidate failing on example X with error "didn't check units." Since then, the candidate was mutated to check units. The recycled trace is wrong — the candidate now handles X. The reflection LLM wastes effort fixing a non-problem.

Mitigation: only recycle traces from the current candidate (depth 1) or parent (depth 2) by default. With depth 2, traces are at most ~3 iterations old, and the candidate has changed by at most 1 mutation. The risk is low.

**Prompt becomes too long.** 3 fresh traces + 5 recycled traces = 8 traces. If each trace is 200 tokens, that's 1600 tokens of trace data. Manageable, but for very verbose evaluators, truncate trace text to a summary.

## Expected Outcome

- **Better diagnoses**: The reflection LLM sees 6-8 failure traces instead of 3. Broader evidence base → more reliable pattern identification → more general fixes.
- **Zero extra evaluation cost**: All traces are recycled from previous iterations.
- **Modest extra prompt cost**: ~500-1000 extra tokens per reflection call.
- **Natural interaction with existing cache**: Cache avoids re-evaluation; trace DB provides diagnostic depth.

## Ablation Design

1. **Standard GEPA, 3-example minibatch** (control)
2. **Evaluation recycling, 3 fresh + 3 historical traces** (depth 2)
3. **Evaluation recycling, 3 fresh + 5 historical traces** (depth 3)
4. **Evaluation recycling, depth 1 only** (only from current candidate — strictest freshness)
5. **Standard GEPA, 6-example minibatch** (matches total trace count of variant 2 but pays the evaluation cost)
6. **Evaluation recycling, all traces (no failure filter)** (also recycle successes — tests whether failure-only is better)

Compare (1) vs (2) for the headline: does trace recycling improve quality? Compare (2) vs (5) to test: is recycling as good as fresh evaluation? Compare (2) vs (4) to test lineage depth sensitivity. Compare (2) vs (6) to test failure-only filtering.

Key metrics: reflection LLM diagnosis quality (measured by val acceptance rate — better diagnoses → better mutations → more acceptances), final val score, and convergence speed.
